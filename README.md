# ZeroMQ Socket (VCV Rack 2 Module & TouchDesigner Bridge) 🧬🔌

Высокопроизводительный сетевой мост с минимальной задержкой между **TouchDesigner** (или внешними Python-скриптами) и **VCV Rack 2** на основе библиотеки **ZeroMQ (ZMQ)**.

Модуль позволяет передавать многоканальные сигналы модуляции (CV) и полифонические MIDI-события (Pitch, Gate, Velocity) по сети без разрывов фазы аудиосигнала, щелчков и пропусков кадров.

---

## 1. Архитектура и пайплайн передачи данных 🛠️

Сетевые операции в аудиодвижках реального времени жестко ограничены по времени. Блокирующие сетевые вызовы в основном DSP-потоке аудиоинтерфейса VCV Rack категорически запрещены, так как они приводят к щелчкам (dropouts) звука.

Архитектура модуля использует **двойную буферизацию** и **выделенный сетевой поток (Worker Thread)**:

```mermaid
graph TD
    TD[TouchDesigner / Python Client] -->|Сетевые пакеты TCP <br> ZMQ PUSH| WT[ZMQ Worker Thread <br> ZMQ PULL socket]
    subgraph VCV Rack Module (ZeroMQSocket)
        WT -->|Копирование memcpy| BF[(Back Buffer)]
        BF -->|Блокировка mutex.try_lock| AF[(Active Buffer)]
        AF -->|DSP Чтение в process| DSP[DSP Audio Thread]
    end
    DSP -->|9x Mono CV| CV_Out[CV Outputs]
    DSP -->|16x Poly Pitch| Pitch_Out[Poly Pitch Output]
    DSP -->|16x Poly Gate| Gate_Out[Poly Gate Output]
    DSP -->|16x Poly Vel| Vel_Out[Poly Velocity Output]
```

1. **TouchDesigner / Python** упаковывает управляющие данные в бинарный пакет и отправляет его через сокет `ZMQ_PUSH` в неблокирующем режиме (`zmq.NOBLOCK`).
2. В **VCV Rack** фоновый поток `ZMQ Worker Thread` слушает сокет `ZMQ_PULL` (таймаут чтения 500мс) и принимает пакеты.
3. Принятые данные мгновенно копируются во внутренний теневой буфер (`Back Buffer`).
4. В основном DSP-потоке VCV Rack (`process()`) проверяется флаг наличия новых данных. Если они есть, то с помощью неблокирующего `mutex.try_lock()` данные копируются в `Active Buffer` и сразу же подаются на физические выходы модуля. Это гарантирует нулевую задержку аудио-потока.

## 2. Спецификация бинарного протокола 📊

Для минимизации оверхеда парсинга (JSON, XML или OSC) передача данных осуществляется в виде сырого бинарного пакета фиксированного размера — **228 байт** (57 чисел типа `float` одинарной точности, 4 байта каждое).

### Структура пакета (`ZmqAudioFrame` в C++):

| Смещение (Байты) | Поле | Тип | Размер | Описание |
| :--- | :--- | :--- | :--- | :--- |
| **0 – 35** | `cv[9]` | `float[9]` | 36 байт | 9 моно-каналов CV (напряжение от -10V до +10V) |
| **36 – 99** | `midi_pitch[16]` | `float[16]` | 64 байта | Высота тона для 16 полифонических голосов (V/Oct, где `0.0` = C4) |
| **100 – 163** | `midi_gate[16]` | `float[16]` | 64 байта | Состояние гейта для 16 голосов (`0.0V` = OFF, `10.0V` = ON) |
| **164 – 227** | `midi_vel[16]` | `float[16]` | 64 байта | Velocity (скорость нажатия) для 16 голосов (от `0.0V` до `10.0V`) |

**Итоговый размер:** 57 * 4 байта = 228 байт.

## 3. Важные принципы полифонии в модуле 🎹

В модульном синтезе полифонический кабель передает несколько независимых каналов (в нашем случае 16). Для корректного звучания синтезатора необходимо соблюдать **правило сохранения высоты тона (Pitch Retention)**:

> [!IMPORTANT]
> При выключении ноты (когда Gate переходит из `10.0V` в `0.0V`), значение **Pitch** для этого канала **НЕ ДОЛЖНО** сбрасываться в `0.0V` (ноту До/C4). Оно должно сохранять последнее сыгранное значение.
> 
> Если сбросить Pitch в 0, то во время фазы затухания огибающей (**Release** в ADSR) генератор VCO мгновенно перестроится на частоту 0V (C4), что создаст паразитный низкочастотный гул («кашу»).

### Реализация Voice Allocation на стороне клиента (Python):
```python
# Инициализация состояния голосов вне цикла отправки
midi_pitch = [0.0] * 16
midi_gate = [0.0] * 16
midi_vel = [0.0] * 16
last_voice = -1

# В цикле при переключении голоса:
if active_voice != last_voice:
    # 1. Тушим гейт предыдущего голоса, сохраняя его pitch
    if last_voice != -1:
        midi_gate[last_voice] = 0.0
    
    # 2. Устанавливаем параметры нового голоса
    midi_pitch[active_voice] = target_pitch
    midi_gate[active_voice] = 10.0
    midi_vel[active_voice] = target_velocity
    
    last_voice = active_voice
```

## 4. Пайплайн интеграции с TouchDesigner (vcv_zmq_bridge.tox) 🎬

Для TouchDesigner создан готовый переносимый компонент **`vcv_zmq_bridge.tox`**, который решает задачи автоматического распределения голосов, упаковки и стабильной сетевой передачи данных.

### 4.1. Установка dependencies (Специфика Python в TouchDesigner)
TouchDesigner на Windows использует собственный встроенный дистрибутив **Python 3.11**, в то время как системный Python обычно новее (например, 3.12). 
Во избежание конфликтов компиляции бинарных файлов сокета (`_zmq.pyd` вызывает ошибку `DLL load failed`), библиотека `pyzmq` для Python 3.11 должна быть установлена локально в папку проекта:

```bash
# Выполните в терминале ОС (установит совместимый 3.11 пакет прямо в папку проекта)
pip install --target "E:\TD Projects\DNA DNB\.py\site-packages" --python-version 3.11 --only-binary=:all: --platform win_amd64 pyzmq
```

При запуске `.tox` автоматически подхватывает эту локальную папку, добавляя её в самое начало путей импорта:
```python
import sys, os
site_pkg = os.path.abspath(os.path.join(project.folder, '.py/site-packages'))
if site_pkg not in sys.path:
    sys.path.insert(0, site_pkg)
```

### 4.2. Параметры компонента (Custom Parameters)
Установив контейнер `vcv_zmq_bridge` в проект, вы получаете доступ к его настройкам на вкладке **ZMQ Bridge**:
* **Active** (Toggle): включает/отключает сетевой мост.
* **Address** (String): IP-адрес VCV Rack (по умолчанию `127.0.0.1`).
* **Port** (Int): Сетевой порт (по умолчанию `5555`).
* **Protocol** (Menu): Режим сокета — `PUSH` (соединение точка-точка, VCV настроен как `PULL`) или `PUB` (широковещательный стриминг, VCV настроен как `SUB`).

*Любое изменение параметров переподключает сетевой сокет на лету без перезапуска TouchDesigner.*

### 4.3. Логика работы Script CHOP (`script_zmq`)
Каждый кадр внутри контейнера запускается Script CHOP, который агрегирует и отправляет данные:
1. **Монофонические CV (Вход 1):** Считывает первые 9 каналов из входного CHOP (`in_cv`) и упаковывает их в `cv[0..8]`.
2. **Прямая полифония (Вход 2, опционально):** Если подключен второй CHOP, скрипт ищет каналы с именами `pitch0..15`, `gate0..15`, `vel0..15` (или короткие `p0`..`p15`, `g0`..`g15`, `v0`..`v15`).
   * *Автомасштабирование:* Если входящие значения гейтов и velocity находятся в диапазоне `0..1` (стандарт TD), скрипт автоматически умножает их на `10.0`, чтобы привести к стандарту VCV Rack (`0..10V`).
3. **Программная полифония:** Если второй вход не подключен, значения полифонии берутся из внутреннего массива расширения (Voice Allocator).
4. **Упаковка:** Сигналы упаковываются через `struct.pack('<57f', ...)` в бинарный пакет размером **228 байт** и отправляются по сокету с флагом `zmq.NOBLOCK`.

### 4.4. Движок полифонии (Python Voice Allocator API)
Для управления высотой звука и огибающими с помощью скриптов (например, через MIDI-ноты или экранную клавиатуру), компонент реализует класс расширения **`VcvBridgeExt`**.

#### Вызов Note On (Включение ноты)
```python
# Синтаксис: PlayNote(note_number, velocity=8.0)
op('vcv_zmq_bridge').PlayNote(60, 8.0) # Сыграть ноту C4 с силой 8V
```
1. **Поиск голоса:** Скрипт сканирует 16 полифонических каналов и занимает первый свободный.
2. **Скейлинг высоты тона:** Преобразует номер MIDI-ноты в вольт-октавный формат: $V = \frac{note - 60}{12}$ (где C4 = 0V, C5 = 1V, C3 = -1V).
3. **Retrigger:** Если нота уже звучит на каком-то канале, гейт сбрасывается в `0V`, принудительно кукается Script CHOP для отправки события снятия ноты, а затем мгновенно зажигается вновь, корректно перезапуская ADSR-огибающую в Rack.
4. **Голосовое вытеснение (Voice Stealing):** Если все 16 каналов заняты, скрипт находит самый старый активный голос, гасит его гейт и назначает канал новой ноте.

#### Вызов Note Off (Выключение ноты)
```python
# Синтаксис: ReleaseNote(note_number)
op('vcv_zmq_bridge').ReleaseNote(60) # Выключить ноту C4
```
* **Pitch Retention:** Сбросит гейт в `0.0V`, переводя ADSR-огибающую в VCV в фазу релиза (**Release**).
* При этом высота тона (`pitch`) и громкость (`velocity`) **сохраняются неизменными** на этом канале. Это критически важно: если сбросить pitch в 0V, нота в фазе затухания резко съедет на C4, испортив микс низкочастотным гулом.

#### Полный сброс
```python
op('vcv_zmq_bridge').ClearAll() # Моментально выключить все гейты
```

## 5. Компиляция и сборка плагина (Windows 11 / MSYS2 MINGW64) 🛠️

Для предотвращения конфликтов C++ ABI (DLL Hell) сборка должна выполняться строго в среде **MINGW64** (с рантаймом MSVCRT). Сборка в UCRT64 вызывает падения VCV Rack при загрузке.

Зависимости ZeroMQ (`libzmq.a` и `libsodium.a`) линкуются **статически**, благодаря чему плагин `plugin.dll` автономен.

### Команды для сборки:
1. Запустите терминал **MSYS2 MinGW 64-bit**.
2. Установите необходимые библиотеки (если не установлены):
   ```bash
   pacman -S mingw-w64-x86_64-zeromq mingw-w64-x86_64-libsodium
   ```
3. Перейдите в корневую папку плагина:
   ```bash
   cd "E:/VSTDEV/JUCE_PROJECTS/VCV MODULES/zmq_plugin"
   ```
4. Выполните сборку и установку:
   ```bash
   make clean
   make install
   ```
   Скрипт скомпилирует `plugin.dll` и упакует его в `Documents/Rack2/plugins/zmq_plugin/`.
